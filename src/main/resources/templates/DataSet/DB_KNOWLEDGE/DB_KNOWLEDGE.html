DB Knowledge

Transaction

  • DB에서 하나의 논리적 기능을 수행하기 위한 작업 단위를 의미
  • 트랜잭션은 데이터의 정합성을 보장하기 위한 기능

Commit : 지금까지 작업한 내용을 DB에 영구적으로 저장

Rollback : 지금까지 작업들을 모두 취소하고 transaction 이전 상태로 되돌린다.

ACID

1. 원자성 (Atomicity)

  • 트랜잭션과 관련된 일이 모두 수행되었거나 되지 않았거나를 보장

2. 일관성 (Consistency)

  • 트랜잭션이 완료된 다음의 상태에서도 트랜잭션이 일어나기 전의 상황과 동일하게 데이터의 일관성을 보장

3. 격리성 (Isolation)

  • 각각의 트랜잭션이 서로 간섭 없이 독립적으로 수행되는 것
    • READ Uncommitted (Dirty Read)
      • 완료되지 않은 트랜잭션 내용을 다른 트랜잭션에서 읽는 문제 발생
    • READ Committed
      • 트랜잭션 작업이 끝난 데이터만 다른 트랜잭션에서 읽을 수 있다.
      • NON-REPEATABLE READ 문제 : 멱등성 관련 데이터 부정합 문제 발생
    • Repeatable READ
      • 다른 트랜잭션에 의해 레코드가 사라지거나 값이 바뀌는 현상을 방지
      • Phantom Read 문제 : 새로운 레코드가 생기거나, 존재하던 레코드가 사라지는 현상
        • MySQL InnoDB의 경우 Undo Log를 버전 관리하기 때문에 Phantom Read 발생하지 않음.
        • Undo Log : rollback을 위해 트랜잭션 시작 시 기존 데이터를 복사해두는 장소
    • Serializable
      • 한 트랜잭션에서 읽고 쓰는 레코드를 다른 트랜잭션에서는 절대 접근 불가
      • 동시 처리 성능도 다른 격리 수준보다 현저히 떨어진다.

4. 지속성 (Durability)

  • 성공적으로 커밋된 트랜잭션은 영구적으로 데이터베이스에 작업의 결과가 저장되어야 한다.

ORM (Object Relational Mapping)

→ 객체 지향 프로그래밍 언어와 관계형 데이터베이스 사이의 호환되지 않는 데이터를 변환하는 프로그래밍 기법

객체를 관계형 DB에서 관리할 때 발생하는 문제점

  1. 애플리케이션 코드보다 SQL문이 더 많아진다.
    • 아무리 설계를 아름답게 하더라도 SQL문을 통해 DB에서 CRUD 작업을 할 수 있기 때문이다.
    • 즉, 단순 반복 작업을 개발자가 직접 해야한다.

  1. 패러다임의 불일치
    • 관계형 DB는 어떻게 데이터를 저장할지 초점이 맞춰진 기술
    • 객체지향 프로그래밍은 메시지 기반의 기능과 속성을 한 곳에서 관리하는 기술
    • 즉, 사상부터 다른 두 패러다임을 중간에서 패러다임을 일치시켜야 함.

N + 1 문제

  • ORM을 사용하는 애플리케이션에서 DB와의 데이터 조회 작업에서 발생 할 수 있는 성능 문제
  • 한 번의 쿼리 실행으로 가져온 데이터를 사용하는 도중 추가로 N번의 쿼리를 실행해야 하는 상황 (성능 저하)

해결 방안

  1. Fetch join : 실행할려는 쿼리와 연관된 데이터를 한 번에 가져와서 쿼리문을 수행한다.
    1. 단점1 : 한 번에 가져와야 하는 데이터가 많을 경우 성능 저하 발생
    1. 단점2 : JPA 구현체에 의존하기 때문에 이식성이 떨어질 수 있다.

  1. Entity Graph : 쿼리 실행 시 필요한 연관 엔티티를 즉시 로드할 수 있다.
    1. 단점1 : NamedEntityGraph를 정의하고 사용해야 하므로 추가 작업 필요
    1. 단점2 : 동적인 쿼리 생성이 제한적

  1. BatchSize : @BatchSize 어노테이션을 사용하여 한 번의 쿼리로 지정된 개수의 연관 엔티티를 로드
    1. 단점1 : BatchSize는 일괄 로딩을 수행하지만, 연관 엔티티가 많은 경우 여전히 N+1 문제가 발생 가능
    1. 단점2 : 적절한 BatchSize 값을 설정해야 한다.

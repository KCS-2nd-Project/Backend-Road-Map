  • 지정한 IP Address에 데이터 전달
  • 패킷 단위로 데이터 전달

IP 프로트콜 한계

  1. 비연결성 : 연결 보장 불가
  1. 비신뢰성 : 패킷 손실 / 순서 보장 X
  1. 프로그램 구분 불가능 : 같은 IP를 사용하는 서버의 프로그램이 2개라면?

TCP / UDP

TCP(Transmission Control Protocol)

  • TCP 3 way handshake (Connect 과정)
    1. SYN
    1. SYN + ACK
    1. ACK
  • 데이터 전달 보증
  • 순서 보장
  • 신뢰 가능한 프로토콜

UDP (User Datagram Protocol)

  • TCP 3 way handshake X
  • 데이터 전달 보증 X
  • 순서 보장 X
  • 데이터 전달 및 순서가 보장되지 않지만, 단순하고 빠름

PORT

만약 한번에 둘 이상의 서버와 연결해야한다면, 패킷의 IP 주소만으로는 한계가 있다!

  • 0 ~ 65535 할당 가능
  • 0 ~ 1023: 잘 알려진 포트, 사용하지 않는 것이 좋음
  • FTP - 20, 21
  • TELNET - 23
  • HTTP - 80
  • HTTPS - 443

DNS (Domain Name System)

IP 주소를 사람이 알기 쉬운 문자로 변환

  • IP 주소는 외우기 어렵다.
  • IP 주소는 변경 가능하다.

HTTP (Hyper Text Transfer Protocol)

  • HTML, TEXT
  • Image, 음성, 영상, 파일 전송
  • JSON, XML (API)

특징

  • 클라이언트 서버 구조
    • Request, Response 구조
    • 클라이언트는 서버에 요청을 보내고 응답을 대기
    • 서버는 클라이언트에 결과를 반환
  • Stateless : 무상태 프로트콜
    • 서버는 클라이언트의 상태 보존을 하지 않는다.
    • 장점 : 스케일 아웃 (확장성 높음)
    • 단점 : 클라이언트는 추가 데이터를 Request에 보내야 함
    • Ex. Stateful 상태라면, 서버와 클라이언트 사이에 연결이 끊기는 경우 모든 내용 초기화

Stateless 정리

  1. 상태 유지 : 클라이언트는 서버에 자신의 상태를 알려줘야 한다.
  1. 무상태 : 무한한 서버 증설 가능

Stateless 한계

  • `로그인`과 같은 상태 정보는 무상태로 할 수 없다.
  • 해결법 : 쿠키 및 세션 등 이용

    → 최소한의 Stateful 이용

Requests

Responses

HTTP Header

  • HTTP 전송에 필요한 부가 정보들

HTTP Body

  • 실제 전송 데이터
  • ex. HTML 문서, JSON …

HTTP 메서드 종류

  • GET : 리소스 조회
  • POST : 요청 데이터 처리
    1. 새 리소스 생성 (등록)
    1. 요청 데이터 처리
    1. 다른 메서드 사용하기 애매한 경우
  • PUT : 리소스 대체
    • 리소스 있으면 대체, 없으면 생성
  • PATCH : 리소스 부분 변경
  • DELETE : 리소스 삭제

HTTP 메서드 속성

  • 안전한 메서드(Safe Methods)
    • 서버에 데이터를 변경하지 않는 요청 메서드
  • 멱등성(Idempotent Methods)
    • 연산을 여러 번 적용하더라도 결과가 달라지지 않는 성질
  • 캐싱 가능(Cacheable Methods)
    • 응답 결과를 서버에 캐싱해서 사용해도 되는 메소드

안전한 메서드

  • GET
  • HEAD

안전하지 않은 메서드

  • POST
  • PUT
  • DELETE
  • PATCH

멱등한 메서드

  • GET
  • HEAD

멱등하지만 안전하지 않은 메서드

  • PUT
  • DELETE

멱등하지 않고 안전하지 않은 메서드

  • POST
  • PATCH

캐싱

  • GET, HEAD, POST, PATCH가 가능하지만 일반적으로 GET, HEAD 정도만 캐시하여 사용

HTTP 상태 코드

  • 2xx (Successful): 요청 정상 처리
    • 200 OK : 요청 성공
    • 201 Created : 요청 성공해서 새로운 리소스 생성
    • 202 Accepted : 요청은 접수됐지만 처리가 완료되지 않음
    • 204 No Content : 요청 성공, 응답 본문에 보낼 데이터 없음
  • 3xx (Redirection): 요청을 완료하려면 추가 행동이 필요
    • 301 Moved Permanently : 요청한 리소스의 URI가 변경되었음을 의미한다. 새로운 URI가 응답에서 아마도 주어질 수 있다 .
    • 308 Permanent Redirect : 301와 유사
      첫 요청에 
      POST가 사용되었다면, 두번째 요청도 반드시 POST를 사용해야 한다.

    • 302 Found : 요청한 리소스의 URI가 일시적으로 변경되었음을 의미한다.
    • 303 See Other : 302와 유사
      리다이렉트 시 요청 메서드가 GET 으로 변경

PRG 패턴 (Post - Redirect - Get)

  • POST는 멱등성을 가지지 않는다.
  • 예를 들어, 인터넷으로 상품을 구매할 때, 새로고침을 한 경우 같은 상품을 여러 번 구매(POST)하게 된다.
  • 따라서 Redirect를 통해 Get 메서드로 변환하여 새로고침을 누르면 조회 기능이 가능하게끔 변경한다.

출처 : https://gofo-coding.tistory.com/entry/PRG-패턴-Post-→-Redirect-→-Get

  • 4xx (Client Error): 클라이언트의 잘못된 요청
    • 400 BadRequest : 잘못된 문법으로 인하여 서버가 요청을 이해할 수 없음을 의미한다.
    • 401 Unauthorized : 클라이언트가 인증하지 않음. (로그인하지 않음)
    • 403 Forbidden : 접근 권한이 없음
    • 404 Not Found : 서버가 요청받은 리소스를 찾을 수 없을 경우

  • 5xx (Server Error): 서버가 정상 요청을 처리하지 못함
    • 500 Internal Server Error : 서버가 처리 방법을 모르는 상황이 발생
    • 502 Bad Gateway : 서버가 요청을 처리하는 데 필요한 응답을 얻기 위해 게이트웨이로 작업하는 동안 잘못된 응답을 수신했음을 의미
    • 503 Service Unavailable : 유지보수를 위해 작동이 중단되거나 과부하가 걸렸을 때
      • Retry-After: HTTP 헤더는 가능하면 서비스를 복구하기 전 예상 시간을 포함 가능

출처 : https://developer.mozilla.org/ko/docs/Web/HTTP/Status

Cookie : 클라이언트가 서버에서 받은 쿠키 저장, HTTP 요청시 서버로 전달

Cookie의 LifeTime

  • 세션 쿠키 : 현재 세션이 끝날 때 삭제됩니다. 즉, 브라우저 종료 전까지 유지
  • 영속 쿠키 : Expires 속성에 명시된 날짜에 삭제되거나, Max-Age 속성에 명시된 기간 이후에 삭제됩니다.

ex. Set-Cookie: id=a3fWa; Expires=Wed, 21 Oct 2015 07:28:00 GMT;

Cookie의 Scope

Path는 Cookie 헤더를 전송하기 위하여 요청되는 URL 내에 반드시 존재해야 하는 URL 경로

만약 Path=/docs이 설정되면, 다음의 경로들은 모두 매칭된다.

  • /docs
  • /docs/Web/
  • /docs/Web/HTTP

Cookie의 보안

Secure

  • http, https를 구분하지 않고 전송
  • Secure 쿠키는 HTTPS 프로토콜 상에서 암호화된(encrypted ) 요청일 경우에만 전송 가능

HttpOnly

  • Cross-site 스크립팅 (XSS (en-US)) 공격 방지
  • HttpOnly 플래그가 설정되지 않은 경우 기본의 쿠키들은 JavaScript로부터 접근 가능

출처 : https://developer.mozilla.org/ko/docs/Web/HTTP/Cookies

Cache

  • 이전에 가져온 리소스들을 재사용함으로써 네트워크 덜 사용
  • Latency와 traffic을 줄여줌으로써 로딩 속도 향상

캐시 시간 초과

  • 캐시 유효 시간이 초과하면, 서버를 통해 데이터를 다시 조회하고 갱신한다.
  • 이때는 네트워크를 사용해야 한다.

Cache-Control: max-age=31536000

캐시 검증

  1. 캐시 유효 시간이 초과해도, 서버의 데이터가 갱신되지 않으면 304 Not Modified (본문을 비워둔 상태로) 응답
  1. 클라이언트는 서버가 보낸 응답 헤더 정보로 캐시의 메타 정보를 갱신
  1. 클라이언트는 캐시에 저장되어 있는 데이터 재활용

→ 결과적으로 네트워크 다운로드가 발생하지만 헤더 정보만 다운로드 가능

  • If-Modified-Since: Last-Modified 사용
  • If-None-Match: ETag 사용
    • ETag : 특정 버전의 리소스를 식별하는 식별자, ETag가 같으면 유지, 다르면 다시 받기
  • 조건이 만족하면 200 OK
  • 조건이 만족하지 않으면 304 Not Modified

캐싱 제어

  • Cache-Control: 캐시 제어
    • Cache-Control: no-cache : 캐시해도 되지만, 유효성 확인을 위해 원 서버로 요청을 보낸다.
    • Cache-Control : no-store : 저장하면 안됨. 매번 요청을 서버에 전송하고 전체 응답을 다운로드한다.
    • Cache-Control: must-revalidate : 오래된 리소스를 사용하기 전, 상태 확인 & 만료된 리소스 시 사용 금지

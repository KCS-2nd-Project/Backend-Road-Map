caching

caching이란

컴퓨팅에서 캐시는 일반적으로 일시적인 특징이 있는 데이터 하위 집합을 저장하는 고속 데이터 스토리지 계층이다. 따라서 이후에 해당 데이터에 대한 요청이 있을 경우 데이터의 기본 스토리지 위치에 액세스할 때보다 더 빠르게 요청을 처리할 수 있다. 캐싱을 사용하면 이전에 검색하거나 계산한 데이터를 효율적으로 재사용할 수 있다.

caching 작동 방법

캐시의 데이터는 일반적으로 RAM(Random Access Memory)과 같이 빠르게 액세스할 수 있는 하드웨어에 저장되며, 소프트웨어 구성 요소와 함께 사용될 수도 있습니다. 캐시의 주요 목적은 더 느린 기본 스토리지 계층에 액세스해야 하는 필요를 줄임으로써 데이터 검색 성능을 높이는 것입니다.

속도를 위해 용량을 절충하는 캐시는 일반적으로 데이터의 하위 집합을 일시적으로 저장합니다. 보통 완전하고 영구적인 데이터가 있는 데이터베이스와는 대조적입니다.

CDN(Cache Delivery Network)

CDN이란

콘텐츠 전송 네트워크는 데이터 사용량이 많은 애플리케이션의 웹 페이지 로드 속도를 높이는 상호 연결된 서버 네트워크이다.

CDN의 목적

CDN의 주 목적은 대기 시간을 줄이거나 네트워크 설계로 인해 발생하는 통신 지연을 감소 시키는 것이다.

따라서 CDN은 웹 사이트 성능을 높이고 핵심 네트워크 인프라를 지원하는 여러 가지 이점을 제공한다.

CDN의 작동원리

CDN은 속도와 연결을 개선하기 위해 다양한 네트워크 사이의 익스체인지 포인트에 서버를 배치한다.

이러한 인터넷 익스체인지 포인트(IXP)는 다른 네트워크에서 발생한 트래픽에 대한 상호 액세스를 제공하기 위해 다양한 인터넷 제공자들이 연결하는 주요 위치이다.

CND의 중요성

  • 페이지 로드 시간 단축

    페이지 로드 시간이 느릴 경우 웹 사이트 트래픽이 감소한다. CDN은 반송률을 줄이고 사용자가 사이트에서 보내는 시간을 늘릴 수 있다.

  • 대역폭 비용 절감

    캐싱 및 기타 최적화를 통해 CDN은 오리진 서버가 제공해야 하는 데이터의 양을 줄여 웹 사이트 소유자의 호스팅 비용을 절감할 수 있다.

  • 콘텐츠 가용성 제고

    한 번에 너무 많은 방문자가 방문하거나 네트워크 하드웨어 오류가 발생하면 웹 사이트가 중단될 수 있다. CDN 서비스는 더 많은 웹 트래픽을 처리하고 웹 서버의 로드를 줄일 수 있다. 또한 하나 이상의 CDN 서버가 오프라인으로 전환되면 다른 운영 서버가 해당 서버를 대체하여 서비스가 중단되지 않도록 한다.

  • 웹 사이트 보안 강화

    CDN은 여러 중간 서버 간에 로드를 분산하여 오리진 서버에 미치는 영향을 줄임으로써 트래픽 급증을 처리할 수 있습니다.

Server Side Caching (Redis, Memcached)

Server Side Caching이란

웹 서버에서 동적인 콘텐츠를 미리 계산하여 저장해 두고, 동일한 요청이 들어왔을 때 빠르게 응답할 수 있도록 하는 기술이다.

Server Side Caching의 특징 및 장점

  • 주로 데이터베이스 쿼리 결과나 API 호출 결과를 캐싱하여 재사용하는 방식으로 이루어진다.
  • 라이언트 단 캐시와 대조적으로, 서버에서 캐싱을 수행한다.
  • 서버의 부하를 줄이고 웹 페이지의 로딩 속도를 향상시킬 수 있다.

Memcached

  • 고성능 캐시 또는 세션 스토어로 사용할 수 있는 사용이 간편한 분산 인 메모리 키-값 스토어이다.
  • 주요 목적은 자주 액세스하는 데이터를 캐시하여 데이터베이스 부담을 줄여 응답 시간을 단축하는 것이다.
  • 사용 편의성과 단순성을 위해 설계된 분산 메모리 캐싱 시스템이며 캐시 또는 세션 저장소로 매우 적합하다.

Redis(REmote Dictionary Server)

  • key,value 구조의 비정형 데이터를 저장하고 관리하기 위한 오픈 소스 기반의 비관계형 데이터 관리 시스템이다.
  • key, value 구조이므로 쿼리가 불필요 하다.
  • 다양한 기능 세트를 제공하는 인메모리 데이터 구조 저장소이다. 캐시, 데이터베이스, 메시지 브로커 및 대기열로 유용하다.

Memcached와 Redis의 공통점

  • 데이터를 메모리에 유지하여 밀리초 미만의 응답 시간을 제공한다.
  • 두 인메모리 데이터베이스 모두 여러 노드에 데이터 배포가 가능하다.
  • Java, Python, C 및 Ruby를 포함한 모든 주요 프로그래밍 언어를 지원한다.
  • 수요가 기하급수적으로 증가할 때 데이터를 처리할 수 있는 높은 확장성을 제공한다.

Client Side Caching

Client Side Caching이란

클라이언트 측 캐싱은 이전에 요청한 파일의 데이터를 브라우저 응용 프로그램이나 기타 클라이언트(중간 네트워크 캐시 등)에 직접 중복 저장하는 기술이다. 클라이언트 캐시는 브라우저가 웹 서버와 통신하지 않고도 파일에 접근할 수 있도록 해 가장 효율적인 캐싱 유형 중 하나이다. 시스템은 파일 요청 응답의 HTTP 헤더를 수정하여 클라이언트 캐싱을 제어한다.

Client Side Caching의 목적

  • 사용자가 이전에 방문한 웹 페이지의 자원(이미지, 스타일시트, 스크립트 등)을 재사용함으로써 웹 페이지 로딩 시간을 줄여 성능을 향상시킨다.
  • 클라이언트 측에서 캐싱된 자원을 사용하면 서버로의 요청을 줄여 서버와의 트래픽을 감소시킬 수 있다.

Client Side Caching의 작동원리

  • 캐싱 헤더: 서버는 응답 헤더에 특정 정보를 포함하여 클라이언트에게 어떤 자원을 얼마나 오랫동안 캐싱할지를 알려준다.
  • 브라우저 저장: 브라우저는 이러한 캐싱 헤더를 기반으로 자원을 캐싱하고, 다음에 같은 자원을 요청할 때 서버에 새로운 요청을 보내지 않고 캐시된 자원을 사용한

Client Side Caching의 중요성

  • 이미 캐시된 자원을 사용함으로써 웹 페이지의 로딩 시간이 크게 감소하므로 사용자는 더 빠르게 웹 페이지를 볼 수 있다.
  • 반복된 요청에 대해 서버에 새로운 요청을 보내지 않아 트래픽이 감소하고 서버 부하를 줄일 수 있다.
  • 서버에서 자원을 전송하는 대신 클라이언트에서 로컬에 저장된 자원을 사용함으로써 서버 비용을 절감할 수 있다.
  • 사용자의 데이터 계획을 고려하여 이미 다운로드한 자원을 다시 사용함으로써 모바일 사용자의 데이터 소모를 감소시킨다.

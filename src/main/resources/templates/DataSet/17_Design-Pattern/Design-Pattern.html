Design Pattern

Design Pattern

Design Pattern이란?

객체 지향 디자인 프로그래밍을 위한 재사용 가능한 클래스 및 객체 설계 패턴이다. GOF(Gang of Four) 라는 23개의 패턴이 가장 일반적이며, 이외에도 더 많은 패턴이 존재한다.

디자인 패턴의 장점

  • 개발자들 간의 원활한 의사소통이 가능하다.
  • 코드를 더 구조적이고 명확하게 하여 소프트웨어의 아키텍처를 파악하기 용이하다.
  • 재사용성과 확장성이 좋아 개발 시간을 단축시킬 수 있다.
  • 설계 변경에 따른 유연성, 이식성이 좋아지기 때문에 유지보수가 용이해진다.

디자인 패턴의 단점

  • 초기 비용이 많이 든다.
  • 객체지향 개발 위주로만 사용 가능하다.
  • 디자인 패턴이 부적절하게 사용될 경우 오히려 문제를 야기할 수 있다.

GoF Design Pattern

GoF(Gang of Four) Design Pattern이란?

'Design Patterns: Elements of Reusable Object-Oriented Software'라는 논문에서 소개된 23가지 디자인 패턴

디자인 패턴은 다음과 같은 세 가지 유형으로 나눌 수 있다.

생성 패턴(Creational)

객체의 생성과 초기화하는 데 사용되는 패턴이다. 인스턴스의 생성 과정을 최대한 숨기고, 객체의 생성을 추상화하여 시스템이 사용하는 클래스에 대한 정보를 캡슐화한다. 특정 객체가 생성되거나 변경되어도 프로그램의 구조에 미치는 영향을 최소화 할 수 있도록 유연성을 제공한다.

  • 싱글톤 패턴(Singleton Pattern): 한 클래스에 대해 오직 한 개의 객체만 존재하도록 보장하도록 제한한다.
  • 팩토리 메소드 패턴 (Factory Method Pattern): 객체를 직접 생성하지 않고 객체를 생성하기 위한 인터페이스를 정의, 이후 서브 클래스가 생성할 객체의 유형을 결정한다.
  • 추상 팩토리 패턴 (Abstract Factory Pattern): 서로 동일한 주제의 객체들을 하나의 집합으로 묶고 이를 생성하는 인터페이스를 제공한다. 이후 구체적인 팩토리 클래스를 통해 객체 생성을 추상화 한다.
  • 빌더 패턴 (Builder Pattern): 복잡한 객체들을 단계별로 생성할 수 있도록 생성 단계를 캡슐화하여 객체 생성 시 해당 과정을 동일하게 이용할 수 있도록 한다.
  • 프로토타입 패턴 (Prototype Pattern): 기존 객체를 복사하여 새로운 객체를 생성한다. 이후 요구 사항에 따라 새 객체에서 변경점을 수정한다.

구조 패턴(Structural)

클래스나 객체를 모아 구조화 시키는 패턴이다. 주로 상속과 컴포지션을 활용하여 새로운 기능을 추가하거나 객체들 간의 인터페이스를 정의한다.

  • 어댑터 패턴 (Adapter Pattern): 호환성이 없는 인터페이스로 인해 함께 동작할 수 없는 클래스들을 함께 동작하도록 하기 위하여, 래퍼(Wrapper) 인터페이스를 제공해 기존의 인터페이스를 변환시켜 사용한다.
  • 브리지 패턴 (Bridge Pattern): 추상화 클래스 계층과 구현 클래스 계층을 분리하여 결합도를 낮추고, 각각 독립적으로 확장할 수 있도록 한다.
  • 컴포지트 패턴 (Composite Pattern): 단일 객체와 복합 객체의 처리 방법이 다르지 않을 경우, 즉 전체-부분 관계로 정의할 수 있을 경우, 클라이언트에서 단일 객체와 복합 객체를 동일하게 다룰 수 있도록 한다.
  • 데코레이터 패턴 (Decorator Pattern): 객체에 독립적, 부가적인 기능을 동적으로 추가하여 객체를 확장한다. 기존 객체의 메소드에 새로운 기능을 추가하거나 오버라이드 함으로써 소스코드를 변경하지 않고 기능을 확장할 수 있다.
  • 퍼사드 패턴 (Facade Pattern): 복잡한 서브시스템에 대한 통합된 인터페이스를 제공하여 시스템을 단순화하고 서브시스템간의 종속성을 감소시킨다.
  • 플라이웨이트 패턴 (Flyweight Pattern): 공유 가능한 객체들을 매번 생성하지 않고 최대한 공유하여 사용함으로써 객체의 메로리 사용을 최소화한다.
  • 프록시 패턴 (Proxy Pattern): 실제 기능을 수행하는 객체를 직접 참조하지 않고 객체에 대한 가상의 객체를 대리자(Proxy)로 제공하여 원본 객체에 대한 접근을 제어한다.

행위 패턴(Behavioral)

객체나 클래스 간의 구체적인 알고리즘을 정의하는 패턴이다. 객체 간의 상호작용과 역할을 중심으로 큰 작업을 여러 개의 객체로 분리하고, 객체들의 상호 작용을 패턴화하여 객체 사이의 결합도를 최소화한다.

  • 책임 연쇄 패턴 (Chain of Responsibility Pattern): 클라이언트의 요청을 처리할 수 있는 처리 객체를 여러개로 나누고 이들을 사슬(chain)처럼 연결한다. 클라이언트에서 요청이 들어오면 요청을 처리하는 객체를 만날 때까지 내부에서 연쇄적으로 요청을 전달한다.
  • 커맨드 패턴 (Command Pattern): 요청 자체를 객체로 캡슐화하여 매개변수화하여 재사용성을 높인다. 해당 패턴을 사용하면 요청을 실행 대기열에 저장하여 재실행하거나, 요청의 실행을 지연시키거나, 요청의 실행을 취소하는 작업을 처리할 수 있다.
  • 인터프리터 패턴 (Interpreter Pattern): 간단한 언어의 문법 규칙을 정의하여 클래스로 표현하고 그 문법을 해석하는 인터프리터를 제공한다.
  • 반복자 패턴 (Iterator Pattern): 반복이 필요한 자료구조(리스트, 트리, 그래프 등)에 대해 모두 동일한 인터페이스를 통해 접근할 수 있도록 함으로써 컬렉션 요소들의 기본 표현을 노출하지 않고 그들을 하나씩 순회할 수 있도록 한다.
  • 중재자 패턴 (Mediator Pattern): 객체 간의 상호작용을 캡슐화하여 중재자라는 하나의 클래스에 위임한다. 객체 간의 직접적인 상호작용을 제한하고 중재자 객체를 통해서만 통신을 처리함으로써 복잡성을 감소시킨다.
  • 메멘토 패턴 (Memento Pattern): 객체의 상태 정보를 가지는 클래스를 따로 생성하여 객체의 세부 사항을 공개하지 않으면서 해당 객체의 상태를 저장하고 복원할 수 있는 기능을 제공한다.
  • 옵저버 패턴 (Observer Pattern): 객체의 상태 변화를 관찰하는 관찰자(Observer)들의 목록을 대상자 객체에 등록해 객체간의 일대다 종속 관계를 정의한다. 대상자 객체의 상태 변화가 있을 때마다 대상자는 이를 감지하고 각 관찰자 객체들에게 통지하고, 관찰자들은 변화를 전달받아 자동으로 조취를 취한다.
  • 상태 패턴 (State Pattern): 객체의 상태를 객체화하고 객체의 내부 상태가 변경될 때마다 객체의 행동을 변경한다. 객체의 내부 상태에 따라 객체는 동일한 루틴에서 다른 기능을 수행한다.
  • 전략 패턴 (Strategy Pattern): 알고리즘군을 정의하고 각각 하나의 클래스로 객체화하여, 실행 중에 알고리즘 전략을 선택하여 객체 동작이 실시간으로 바뀔 수 있도록 한다.
  • 템플릿 메서드 패턴 (Template Method Pattern): 여러 클래스에서 공통으로 사용하는 메소드를 템플릿화 하여 상위 클래스에서 정의한다. 하위 클래스에서는 상위 클래스에서 정의된 골격을 기반으로 알고리즘의 특정 단계들을 오버라이드하여 구체적으로 구현한다.
  • 방문자 패턴 (Visitor Pattern): 객체의 구조로부터 알고리즘을 분리하여 별도의 방문자(visitor) 클래스로 정의한다. 방문자 클래스는 객체 구조의 각 클래스를 방문하면서 그 클래스에서 수행할 작업을 수행한다.

Domain Driven Design

Domain Driven Design이란?

소프트웨어 개발 프로세스에서 핵심 도메인에 초점을 맞추어 문제를 해결하는 개발 방법론이다. 소프트웨어의 연관된 부분들을 연결하여 계속해서 진화하는 모델을 만들어 나가면서 복잡한 비즈니스 도메인을 이해하고, 해당 도메인을 기반으로 소프트웨어 시스템을 구축하는 데 중점을 둔다.

Domain Driven Design 단계

  1. Strategic Design(전략적 설계)
    • 도메인 전문가 및 기술팀이 유비쿼터스 언어를 통해 지식을 공유 및 이해하고 이를 기준으로 도메인을 탐색한다.
    • 탐색된 도메인을 여러 Bounded Context로 나누어 각 모델을 구분하여 관리할 수 있도록 한다.
    • Context Mapping을 통해 각 Bounded Context같의 상호작용을 정의하고, 이를 통해 전체적인 프로젝트의 구조를 결정한다.
  1. Tactical Design(전술적 설계):
    • 전략적 설계에서 도출된 Bounded Context를 기반으로 Domain Driven Design의 개념과 패턴을 활용하여 세부적인 도메인 모델링을 진행한다.
    • 도메인 로직을 구성하고 구현함으로써 프로젝트 구조를 구체화한다.

Test Driven Design

Test Driven Design이란?

소프트웨어 개발 프로세스에서 매우 짧은 시간 개발 사이클을 반복하는 개발 방법론이다. 작은 단위의 테스트 케이스를 작성하고 이를 통과하는 코드를 추가하는 단계를 반복하여 프로그램을 구현한다.

Test Driven Design 개발 주기

  1. 테스트 작성(Red)
    • 구현하려는 기능에 대한 실패하는 테스트 코드를 작성한다.
    • 작성한 테스트 코드를 실행하여 테스트가 실패하는 것을 확인한다.
  1. 구현(Green)
    • 테스트 코드를 성공시키기 위한 실제 코드를 작성한다.
    • 테스트를 실행하여 기능이 정상적으로 작동하는지 확인한다.
  1. 리팩터링(Refactor)
    • 리펙터링을 통해 코드의 가독성을 향상시키고 중복 코드를 제거하여 코드를 개선하고 품질을 향상시킨다.
    • 리펙터링한 코드에 대해 기존의 테스트를 실행하여 리팩터링 과정에서 문제가 발생하지 않고 기능이 여전히 정상적으로 작동하는지 확인한다.

Test Driven Design 장점

  • 디버깅 시간을 단축할 수 있다.
    에러가 생겼을 경우 자동화된 단위테스트를 통해 특정 버그를 손 쉽게 찾아낼 수 있다.
  • 재설계 시간을 단축할 수 있다.
    설계 단계에서 테스트 시나리오를 먼저 작성하기 때문에 개발자가 무엇을 해야하는지 분명히 정의할 수 있다. 또한, 다양한 다양한 예외 사항을 먼저 생각해 봄으로써 추후 설계가 변경되는 일을 방지할 수 있다.
  • 유지보수가 용이하다.
    코드 변경이나 업데이트 시 기존의 기능이 올바르게 작동하는 지 자동화된 단위 테스트를 기반으로 빠르게 확인할 수 있어 유지보수가 용의하다.

Test Driven Design 단점

  • 생산성 저하가 발생한다.
    처음부터 2개 이상의 코드를 반복하여 작성하고, 중간중간 테스트를 진행하며 코드를 수정해야 하기 때문에 개발 시간이 증가한다.

SOLID

SOLID란?

객체 지향 프로그래밍 및 설계의 다섯 가지 기본 원칙이다. SOLID 원칙을 적용함으로써 유지보수와 확장이 쉬운 프로젝트를 개발할 수 있다. 또한 불필요한 복잡성을 제거함으로써 프로젝트 개발의 생산성을 높일 수 있다.

단일 책임 원칙 - SRP (Single Responsibility Principle)

하나의 클래스는 하나의 책임(기능)만 가져야 한다.

만일 하나의 클래스가 여러 책임을 가진다면, 한 기능의 변경이 발생하였을 때 수정해야 할 코드의 양이 많아질 뿐만 아니라 다른 기능에 관한 코드가 망가질 위험이 높다.

개방 폐쇄 원칙 - OCP (Open Closed Principle)

소프트웨어 요소는 확장에는 열려 있어야하고, 수정에는 닫혀 있어야 한다.
  • 확장에 열려 있어야 한다.
    요구 사항이 변경 되었을 때 유연하게 새로운 코드를 추가함으로써 기능을 확장할 수 있다.
  • 수정에 닫혀있어야 한다.
    기존의 코드를 수정하지 않아도 새로운 기능을 추가하거나 기능을 변경할 수 있다.

리스코프 치환 원칙 - LSP (Liskov Substitution Principle)

하위 타입은 언제나 상위 타입으로 변경할 수 있어야 한다.

즉, 상속 관계에서 하위 클래스가 상위 클래스의 기능을 동일하게 제공해야 한다.

  • 확장에 열려 있어야 한다.
    요구 사항이 변경 되었을 때 유연하게 새로운 코드를 추가함으로써 기능을 확장할 수 있다.
  • 수정에 닫혀있어야 한다.
    기존의 코드를 수정하지 않아도 새로운 기능을 추가하거나 기능을 변경할 수 있다.

인터페이스 분리 원칙 - ISP (Interface Segregation Principle)

하나의 일반적인 인터페이스보다 여러 개의 구체적인 인터페이스가 낫다.

클라이언트는 자신이 사용하지 않는 메소드와 의존 관계를 맺거나 클라이언트가 사용하지 않는 인터페이스에 의해 영향을 받아서는 안된다. 따라서 인터페이스 내에 메소드를 최소한으로 하여 하나의 역할에 집중하도록 하고, 클라이언트의 목적과 용도를 기준으로 적합한 적합한 인터페이스 만을 제공하여야 한다.

의존 역전 원칙 - DIP (Dependency Inversion Principle))

구체적인 것이 추상화된 것에 의존해야 한다. 자주 변경되는 구체 클래스에 의존하지마라.

의존 관계를 맺을 때 변화하기 쉬운 것, 자주 변화하는 것보다는 변화가 어렵거나 거의 변화가 없는 것에 의존해야 한다. 즉, 구체적인 클래스보다 상위 클래스, 인터페이스, 추상 클래스와 같이 변하지 않을 가능성이 높은 클래스와 관계를 맺어야 한다.

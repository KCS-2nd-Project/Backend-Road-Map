Web Server

Web Server 란?

→ 클라이언트의 HTTP 요청을 웹 서버는 받고 HTML 문서와 같은 정적
콘텐츠를 사용자에게 전달해주는 역할

  • 대표적인 웹 서버 : Apache, Nginx, IIS

WAS (Web Application
Server) 란?

→ 클라이언트의 HTTP 요청을 웹 서버는 받으면 WAS 서버에서는 비즈니스
로직을 처리하여 사용자에게 동적인 콘텐츠를 전달해주는 역할. 주로 DB
서버와 함께 수행

  • 대표적인 WAS : Tomcat, JBoss, Jeus

→ Web Server는 정적인
데이터 처리를 위한 서버로 빠르고 안정적

→ WAS는 동적인 데이터
처리를 위한 서버로 사용자의 데이터를 조작, 통신이 필요한 경우 사용

장점단점
Apache(static)1. 처리 속도가 빠르다.
2. 구조가 단순함

3. 트래픽 과부하에 강함
1. 정적인 데이터만 처리 가능
2. 다른 서비스와 상호작용 불가
Tomcat(dynamic)1. 데이터 흐름이 유동적1. 아파치에 비해 속도가 늘다.
2. 부가적인 비용 발생

3. 트래픽 과부하에 약함

  • 참고 :https://httpd.apache.org/

Nginx

  • 비동기 I/O 처리 방식을 사용하여 적은 메모리
    사용량으로 높은 성능을 제공
  • Reverse Proxy 사용 가능 : 원본 서버 앞에 프록시를
    위치하여 어떤 클라이언트도 원본 서버와 직접 통신하지 못하도록 막음. 또한
    리버스 프록시를 통해 user의 길을 분배하여 트래픽 부하를 줄일 수 있다.
    이를
    로드 밸런싱 이라고 한다.

24시간 365일 무중단 배포
서비스

  • 출처 : <스프링 부트와 AWS로 혼자 구현하는 웹 서비스>, 저자 :
    이동욱
  1. 엔진엑스는 80(http), 443(https) 포트를 할당한다.
  1. 스프링 부트1은 8081 포트로 실행한다.
  1. 스프링 부트2은 8082 포트로 실행한다.

① 사용자는 서비스 주소로 접속 (80 또는 443 포트)

② Nginx는 요청을 받아 스프링 부트로 요청 (8081포트로 가정)

③ 8082는 Nginx와 연결된 상태가 아니기 때문에 요청받지 못한다.

① 배포하는 동안에도 서비스는 중단되지 않는다. (8081 서버와 연결되어
있기 때문!!)

② 배포가 끝난 후 스프링 부트2(8082포트)가 정상 작동하는지 확인

③ 스프링 부트2(8082포트)가 정상 작동한다면, nginx reload를 통해 8081
대신 8082를 바라보게 한다.

④ nginx reload는 0.1초 이내에 완료!
